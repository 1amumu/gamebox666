# 项目记忆

更新时间：2026-05-13
工作目录：`D:\gongzuo\api\商户后台`

## 1. 项目定位

这是一个中文商户后台 / API ADMIN 的静态原型项目，主要用于演示和迭代以下后台业务：

- 仪表盘运营概览
- 玩家账号与会员详情
- 游戏记录、今日玩家盈亏
- 数据统计与报表
- 商户报表
- 游戏配置 / 点控
- API 配置
- 系统管理、角色管理、操作日志

项目重点是“可直接预览、可快速改 UI、可持续按截图迭代”，不是完整的工程化前后端应用。

## 2. 技术结构

当前项目以原生静态文件为主：

- `*.html`：各业务页面
- `admin.css`：全局样式和页面公共视觉规则
- `sidebar.js`：左侧菜单统一渲染与折叠交互
- `common.js`：轻量公共脚本
- `account-detail-shared.js`：共享会员详情弹层
- `serve-1181.js`：本地静态服务脚本

当前没有 `package.json`、没有构建流程、没有模块化打包。修改后通常直接在浏览器刷新查看结果。

## 3. 本地运行方式

当前项目默认通过 1181 端口预览：

- 启动脚本：`serve-1181.js`
- 监听地址：`http://127.0.0.1:1181/`

脚本本质是一个轻量静态文件服务，适合直接预览本目录下的 HTML 页面。

## 5. 关键页面记忆

### 5.1 玩家账号

文件：`player-account.html`

这是当前最重的会员详情演示页之一，已经支持：

- 平台 ID 点击打开会员详情
- 会员详情弹层支持全屏 / 还原
- 标签页切换
- 同设备账号入口
- 注册 IP、登录 IP 的 hover 弹出展示
- 同设备账号数量展示
- 账号状态、账号标签、商户信息、RTP、投注、输赢等字段

当前这个页面自己的会员详情逻辑仍然保留在页内脚本中。

### 5.2 游戏记录

文件：`player-game-records.html`

当前已按最近迭代补充了以下业务点：

- 订单编号前增加玩家 ID
- 玩家 ID 复用“用户账号”里的平台 ID 语义
- 玩家 ID、订单编号、牌局编号支持快速复制
- 触发方式只保留两个状态：`普通投注`、`免费旋转`
- 会员详情中支持 `每日数据`
- 每日数据按日展示近一个月，可选日期区间
- 缺失日期补 0 展示
- 已加入 `控制状态`，区分 `点控中` 与 `正常`

当前该页已经接入 `account-detail-shared.js`，点击玩家 ID 会打开共享会员详情弹层。

### 5.3 今日玩家盈亏

文件：`player-today-profit-loss.html`

这是另一张高频数据页，重点特征包括：

- 平台 ID、游戏、日期等筛选
- 游戏选择器
- 表格分页和每页条数切换
- 平台 ID 支持复制

当前该页也已接入 `account-detail-shared.js`，点击平台 ID 会打开与“用户账号”同一套会员详情体验。

### 5.4 商户报表

文件：`merchant-report.html`

这是当前浏览器上下文中正在查看的页面之一。它属于典型报表页，后续改动默认遵循统一后台报表风格：

- 筛选区紧凑
- 数据表优先
- 导出入口轻量
- 不做营销式大卡片堆叠

### 5.5 系统管理 / 操作日志

文件：`system-operation-log.html`

已新增“操作日志”页面，位于“角色管理”下方。当前页面目标是记录每个后台管理员的操作行为，重点包含：

- 管理员账号
- 角色
- 操作类型
- 所属模块
- 操作对象
- 修改内容
- 登录 IP
- 操作 IP
- 时间
- 结果

涉及修改类操作时，需要能看出改前改后内容。

## 6. 共享会员详情弹层

文件：`account-detail-shared.js`

这是最近新增的重要公共脚本，目的是把“用户账号”中的会员详情能力复用到其他页面，避免每个页面重复维护一套弹层。

当前共享脚本包含：

- `PROFILE_MAP`
- `BET_RECORDS`
- `SAME_DEVICE_MAP`
- 共享会员详情弹层注入
- 标签页切换
- 投注记录筛选
- 同设备账号展示
- 复制按钮绑定
- 注册 IP / 登录 IP hover 内容

当前自动生效页面：

- `player-game-records.html`
- `player-today-profit-loss.html`

共享弹层点击绑定策略：

- 使用捕获阶段监听 ID 点击
- 遇到 `[data-copy]` 时跳过，避免复制按钮触发弹层
- 在游戏记录页中会拦截旧点击逻辑，避免双弹层冲突

## 7. 当前视觉与交互偏好

这是项目已经反复确认过的默认规则：

- 偏后台工作台风格，不做营销页风格
- 白底、浅灰边框、清晰分区、信息密度适中
- 优先 PC 宽屏展示，但也要避免常见尺寸下错位
- 筛选栏尽量一行排开，不拥挤时不主动换行
- 交互必须能用，不能只改文案不补行为
- 复制、悬浮、弹层、分页、筛选都要有真实反馈
- 控件之间不能互相遮挡，尤其是筛选栏、hover、弹层、左侧菜单

## 8. 用户协作偏好

这是非常重要的协作记忆，后续默认继续沿用：

- 用户提需求通常很直接，往往是短句命令式反馈，默认直接执行
- 用户偏好“边看边改”，希望每轮都能在浏览器马上看到变化
- 用户常通过 in-app browser 截图、批注、圈选局部区域来提需求
- 用户很在意后台页面是否实用、清楚、顺手，不喜欢花哨装饰
- 用户希望我主动把项目规则和迭代结果记到 `.md` 文件中，减少重复解释
- 当页面已有明确视觉参照时，应优先复刻已有页面的交互和结构，而不是重新发明一套

## 9. 维护约定

后续修改时默认遵守：

1. 菜单相关只改 `sidebar.js`
2. 公共视觉优先落在 `admin.css`
3. 页面级业务逻辑尽量留在对应页面，跨页复用逻辑再抽公共脚本
4. 能复用现有交互时优先复用，尤其是会员详情、游戏选择器、复制按钮、筛选区
5. 做完后优先验证页面能正常打开，必要时检查脚本语法和本地服务状态
